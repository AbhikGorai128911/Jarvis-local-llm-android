JARVIS LOCAL LLM (V2 ARCHITECTURE OVERVIEW)
1. PROJECT SUMMARY
JARVIS V2 is a hybrid Android-based local assistant built on:
Termux environment (Android Linux layer)
llama.cpp local LLM server (http://127.0.0.1:8080⁠�)
Rule-based intent classification system
Capability-based device control layer
Modular command execution architecture
Goal: Build an offline-first assistant capable of:
system control
device control
web actions
local reasoning via LLM
structured extensibility for future AI upgrades
2. SYSTEM ENTRY FLOW
main.py flow:
User Input → intent_classifier.py → router.py → executor.py → command modules OR device capabilities OR LLM fallback
3. INTENT CLASSIFICATION SYSTEM
File: reasoning/intent_classifier.py
Current intents:
system → app launch / Android intents
web → browsing/search
music → media playback (strictly limited scope)
device → hardware/system toggles (torch, wifi, bluetooth, volume, brightness)
files → filesystem operations
browser → app/web shortcuts
memory → factual / Q&A style queries
communication → SMS / messaging
unknown → fallback to LLM
Key fix:
Removed overlapping music/device conflicts
Volume and hardware controls moved into device intent
Priority: device > music > system
4. ROUTING LAYER
File: core/router.py
Responsibilities:
Call intent classifier
Check execution permission via decision engine
Route to executor
Trigger LLM fallback for unknown intents
5. EXECUTION LAYER
File: control/executor.py
Dispatch system:
system_execute → Android app intents
web_execute → browser/search
music_execute → playback control
file_execute → filesystem operations
memory_execute → stored knowledge queries
device_execute → hardware control via capabilities system
LLM fallback → llama.cpp inference
6. DEVICE CAPABILITY SYSTEM (V2 CORE FEATURE)
Files:
capabilities/capabilities.py
capabilities/resolver.py
capabilities/executor.py
Purpose: Convert natural language device commands into structured operations.
Examples:
"turn on torch" → torch ON via termux-torch
"turn on wifi" → open Android WiFi settings
"volume 10" → termux-volume music 10
"bluetooth on" → Bluetooth settings intent
Unsupported:
mobile data toggle (Android restriction)
dark mode toggle (OEM restriction)
direct WiFi control (system-level restriction)
7. TERMUX API INTEGRATION
Active APIs:
termux-torch
termux-volume
termux-battery-status
termux-sensor
termux-contact-list
termux-location (permission required)
termux-sms-send (permission required)
Restricted or permission-bound:
brightness control (WRITE_SETTINGS required)
SMS read access
location access
8. ANDROID INTENTS LAYER
Used for system UI control:
android.settings.BLUETOOTH_SETTINGS
android.settings.WIFI_SETTINGS
android.settings.DISPLAY_SETTINGS
android.settings.ACCESSIBILITY_SETTINGS
Executed via:
am start -a <ACTION>
9. LLM INTEGRATION LAYER
File: llm/local_llm.py
Backend:
llama.cpp server running locally
Mode types:
On-demand mode (default)
Persistent mode (planned upgrade)
Used when:
intent = unknown
complex reasoning required
fallback when rule system fails
10. CURRENT LIMITATIONS
Rule-based intent classifier (no ML scoring yet)
No command parsing system (e.g., "volume 10" is partially handled)
Limited Android control due to OS restrictions
No long-term action memory graph
No adaptive learning system
11. SYSTEM DESIGN PHILOSOPHY
Core principle:
Deterministic control first, AI fallback second.
Meaning:
System always tries structured execution first
LLM is only used when deterministic routing fails
12. V3 ROADMAP
Next evolution:
Intent scoring system (replace keyword matching)
Command parser (structured action extraction)
Persistent LLM orchestration layer
